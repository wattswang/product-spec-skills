# Spec 转测试计划

将产品需求文档（Spec/PRD）转化为结构化的测试策略、测试用例和质量验证计划。应用 ISTQB 测试设计技术和 ISO 25010 质量模型，确保需求覆盖完整。结果输出到 Obsidian，并可选同步到 Coding 平台创建测试任务。

## 输入

`$ARGUMENTS` 支持以下来源（自动识别）：

| 输入类型 | 示例 | 识别方式 |
|---------|------|---------|
| 本地文件 | `/path/to/spec.md` 或 `D:\docs\spec.md` | 包含路径分隔符且为文件扩展名 |
| Obsidian 笔记 | `obsidian://Specs/my-feature` 或 `ob:Specs/my-feature` | 以 `obsidian://` 或 `ob:` 开头 |
| Notion 页面 | `https://www.notion.so/...` | 包含 `notion.so` |
| Coding 链接 | `https://xxx.coding.net/p/xxx/defect/123` | 包含 `coding.net` |
| 图片/设计稿 | `/path/to/mockup.png` 或 Figma URL | 图片扩展名或 `figma.com` |
| 直接描述 | 任意自然语言文本 | 以上均不匹配时 |

如果 `$ARGUMENTS` 为空，询问用户提供 Spec 来源。

**额外输入**：如果已经执行过 `/spec-to-tasks`，可以同时提供任务分解文档的路径，以便更精确地生成测试用例。

## 执行步骤

### Step 1: 获取 Spec 内容

根据输入类型获取 Spec 内容（方式同 spec-review 的 Step 1）：

- **本地文件**：使用 `Read` 工具读取
- **Obsidian 笔记**：使用 `ToolSearch` 查找 `obsidian_read_note`，然后读取
- **Notion 页面**：使用 `ToolSearch` 查找 `notion-fetch`，然后获取
- **Coding 链接**：提取 `project_name` 和 `issue_code`，使用 `get_defect` 获取
- **图片**：使用 `Read` 工具查看
- **直接描述**：直接使用文本

同时检查 Obsidian 中是否存在已有的任务分解文档（`Specs/Tasks/{spec-name}-tasks.md`），如果存在则一并读取。

### Step 2: 需求分析

提取 Spec 中所有可测试的需求点：

1. **功能需求**：每个用户故事和验收标准
2. **非功能需求**：性能、安全、可用性等
3. **业务规则**：数据校验、流程约束、权限控制
4. **接口契约**：API 输入输出、错误码、数据格式

为每个需求点分配编号：`REQ-001`, `REQ-002`, ...

### Step 3: 测试策略制定

#### 3.1 测试类型分层

| 测试层级 | 覆盖范围 | 覆盖率目标 | 框架建议 |
|---------|---------|-----------|---------|
| 单元测试 | 独立函数、工具类、业务逻辑 | 行覆盖 >80% | JUnit/Jest/pytest |
| 集成测试 | API 接口、数据库操作、服务间调用 | 接口覆盖 100% | MockMvc/Supertest |
| E2E 测试 | 关键用户流程 | 核心路径 100% | Playwright/Cypress |
| 性能测试 | 高并发场景、大数据量 | 按 SLA 要求 | JMeter/k6 |
| 安全测试 | 认证、授权、注入、XSS | OWASP Top 10 | OWASP ZAP |

#### 3.2 ISTQB 测试设计技术

对每个需求应用适当的测试设计技术：

- **等价类划分**：将输入域划分为有效类和无效类
- **边界值分析**：测试边界条件（最小值、最大值、边界±1）
- **决策表**：多条件组合场景
- **状态转换**：有状态流转的功能（如订单状态、审批流程）
- **经验法探索**：基于常见 Bug 模式的探索性测试

#### 3.3 ISO 25010 质量维度

评估以下质量特性的测试需求（标记需要/不需要）：

| 质量维度 | 是否需要 | 测试重点 |
|---------|---------|---------|
| 功能适合性 | ✅ | 功能完整性、正确性 |
| 性能效率 | {根据Spec} | 响应时间、吞吐量、资源利用 |
| 兼容性 | {根据Spec} | 浏览器兼容、API 版本兼容 |
| 易用性 | {根据Spec} | 可访问性、用户体验 |
| 可靠性 | {根据Spec} | 容错、可恢复性 |
| 安全性 | ✅ | 认证、授权、数据保护 |
| 可维护性 | {根据Spec} | 代码质量、可测试性 |
| 可移植性 | {根据Spec} | 环境适配 |

### Step 4: 测试用例生成

为每个需求生成测试用例，使用 Given/When/Then 格式：

#### 4.1 正向路径（Happy Path）

```gherkin
TC-001: {测试用例标题}
关联需求: REQ-001
优先级: P0
前置条件: {Given}

Given {初始状态}
When {用户操作}
Then {预期结果}
```

#### 4.2 异常路径（Error Path）

覆盖以下异常场景：
- 参数缺失 / 参数格式错误
- 权限不足 / 未认证
- 数据不存在 / 数据冲突
- 网络超时 / 服务不可用
- 并发操作冲突

#### 4.3 边界条件（Boundary）

覆盖以下边界场景：
- 空值 / 空字符串 / 空数组
- 最大长度 / 最大数值
- 特殊字符 / Unicode / Emoji
- 分页边界（第一页、最后一页、超出范围）

### Step 5: 测试用例与需求追踪矩阵

建立需求到测试用例的映射关系：

| 需求编号 | 需求描述 | 测试用例 | 覆盖状态 |
|---------|---------|---------|---------|
| REQ-001 | {需求} | TC-001, TC-002, TC-003 | ✅ 已覆盖 |
| REQ-002 | {需求} | TC-004 | ✅ 已覆盖 |
| REQ-003 | {需求} | — | ❌ 未覆盖 |

**覆盖率统计**：
- 需求覆盖率：已覆盖需求数 / 总需求数
- 测试用例总数：N（正向 X + 异常 Y + 边界 Z）

### Step 6: 输出到 Obsidian

使用 `ToolSearch` 查找 `obsidian_update_note` 工具，将测试计划写入 Obsidian。

**输出路径**：`Specs/Tests/{spec-name}-test-plan.md`

**输出模板**：

```markdown
---
tags: [test-plan, {project-name}]
date: {YYYY-MM-DD}
spec-source: {来源类型和路径}
total-requirements: {N}
total-test-cases: {N}
requirement-coverage: {百分比}
---
```

### Step 7: 同步到 Coding（可选）

在 Obsidian 输出完成后，询问用户是否要将测试任务同步到 Coding 平台。

如果用户确认：

1. 使用 `ToolSearch` 查找 `coding` 相关工具
2. 按测试层级创建任务，每个层级一个 defect：
   - `[测试] 单元测试 — {Spec名称}`：包含该层级所有测试用例清单
   - `[测试] 集成测试 — {Spec名称}`：包含该层级所有测试用例清单
   - `[测试] E2E测试 — {Spec名称}`：包含该层级所有测试用例清单
   - `[测试] 性能测试 — {Spec名称}`（如适用）
   - `[测试] 安全测试 — {Spec名称}`（如适用）
3. 参数映射：
   - `title` ← 测试任务标题
   - `description` ← 测试用例清单 + 验收标准 + 覆盖率目标
   - `priority` ← 2（高优先级）
   - `assignee_ids` ← 用户选择的测试人员
   - `iteration_id` ← 目标迭代 ID
4. 输出创建结果汇总

## Rules

- **需求全覆盖**：Spec 中的每个需求点必须至少有一个对应的测试用例，追踪矩阵中不允许有 ❌
- **不跳过异常**：每个功能至少覆盖 2 个异常路径
- **不跳过边界**：涉及数值、长度、分页的功能必须有边界测试
- **Given/When/Then**：所有测试用例必须使用 GWT 格式，步骤必须具体可执行
- **优先级标注**：每个测试用例必须标注优先级（P0 核心 / P1 重要 / P2 补充）
- **中文输出**：测试计划使用中文，Gherkin 关键词保留英文
- **创建前确认**：在 Coding 上创建测试任务前，必须先展示任务列表并获得用户确认
